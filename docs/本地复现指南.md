# 本地复现指南

## 1. 准备环境

推荐版本：

- JDK 8
- Maven 3.6+
- Docker Desktop

课程项目是 Spring Boot 2.3.12 + MySQL 5.7 + Redis。仓库根目录已经准备了 `docker-compose.yml`，MySQL root 密码为 `123`，Redis 密码为 `123321`，与课程配置保持一致。

当前已启动的本地学习环境使用工作区专用 MySQL `127.0.0.1:3307` 和工作区专用 Redis `127.0.0.1:6380`；二者密码都是 `l798267901`，对应配置在 `application-local.yaml`。

## 2. 启动 MySQL 和 Redis

在仓库根目录执行：

```powershell
docker compose up -d
```

第一次启动时，MySQL 会自动导入：

```text
resources/db/hmdp.sql
```

检查容器：

```powershell
docker compose ps
```

## 3. 初始化 Redis Stream 消费者组

完整实现版的秒杀订单使用 Redis Stream。第一次运行前执行一次：

```powershell
docker exec -it hmdp-redis redis-cli -a 123321 XGROUP CREATE stream.orders g1 0 MKSTREAM
```

如果提示 `BUSYGROUP`，说明已经创建过，可以忽略。

## 4. 运行后端

完整实现版：

```powershell
cd backend\hm-dianping-complete
mvn spring-boot:run -Dspring-boot.run.profiles=local
```

起始版：

```powershell
cd backend\hm-dianping-starter
mvn spring-boot:run -Dspring-boot.run.profiles=local
```

启动成功后，后端监听：

```text
http://127.0.0.1:8081
```

## 5. 运行前端

课程前端默认把 API 请求发到 `/api`，所以建议使用 Nginx 做静态资源和反向代理。原始 Nginx 包已备份在：

```text
resources/runtime/nginx-1.18.0.zip
```

如果你想直接用课程包：

1. 解压 `resources/runtime/nginx-1.18.0.zip`
2. 确认 `conf/nginx.conf` 中 `/api` 代理到 `http://127.0.0.1:8081`
3. 启动 `nginx.exe`
4. 浏览器访问 `http://localhost:8080`

当前工作区已经启动了一个独立 Nginx 副本，端口是：

```text
http://127.0.0.1:8082
```

如果只是看页面结构，可以直接打开：

```text
frontend/hmdp/index.html
```

但直接打开 HTML 时没有 `/api` 代理，后端接口不会正常联通。

## 6. 常用验证动作

发送验证码：

```powershell
curl "http://127.0.0.1:8081/user/code?phone=13800138000"
```

验证码会打印在后端日志里。登录后得到 token，前端会把 token 放入 `authorization` 请求头。

导入附近商户 GEO 数据：

```powershell
cd backend\hm-dianping-complete
mvn -Dtest=HmDianPingApplicationTests#loadShopData test -Dspring.profiles.active=local
```

秒杀压测脚本：

```text
resources/jmeter/秒杀抢购.jmx
```

## 7. 常见问题

`RedisConnectionFailureException`：

- 确认 `docker compose ps` 中 Redis 正常
- 确认运行后端时加了 `local` profile
- 确认 Redis 密码是 `123321`

`Access denied for user 'root'`：

- 确认 MySQL 容器是由本仓库 `docker-compose.yml` 创建
- 如果之前已有同名 volume，可执行 `docker compose down -v` 后重新启动；这会删除本项目容器数据

秒杀接口返回正常但没有落库：

- 确认已经创建 `stream.orders` 的 `g1` 消费者组
- 查看后端日志是否有 Stream 消费异常
